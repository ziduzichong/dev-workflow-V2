## Scope Gate（使用前必须判定）

### 50行法则

| 改动规模 | 判定 | 执行方式 |
|---------|------|---------|
| ≤50 行，单文件 | **不启动工作流** | 直接编码 + 编译验证 |
| ≤50 行，多文件 | **简化流程** | 仅阶段1(规划) + 阶段3(编码) + 阶段5(审查) |
| 50~200 行 | **标准流程** | 执行 7 阶段，可跳过阶段2/4/6 |
| ≥200 行 或 多模块 | **完整流程** | 严格执行全部 7 阶段，不可跳过 |

### 适用
- 涉及 ≥3 个文件的改动
- 预计代码量 ≥50 行
- 需要架构设计的多模块项目
- 从零开始的新功能/新项目

### 不适用
- 单文件小修改（修 typo、改常量、加注释）
- 单函数 bug 修复 → 直接使用阶段4 系统化调试流程
- 纯咨询/解释性问题

**如果判定为不适用，告知用户并建议直接用普通对话完成。**

**如果判定为简化流程，告知用户将跳过哪些阶段并获确认后再执行。**

## 苏格拉底追问（Scope Gate 判定通过后立即执行）

进入阶段1 之前，对需求进行深度追问，每次一个问题：

1. **目标追问** — "这个功能最终要解决什么问题？谁来使用？使用频率多高？"
2. **边界追问** — "什么情况下不应该触发这个功能？最大数据量/并发量是多少？"
3. **约束追问** — "有哪些技术限制（内存/Flash/延迟/依赖）？有哪些非技术限制（时间/合规/兼容性）？"
4. **验收追问** — "怎么判断这个功能做完了？验收标准是什么？"

**原则：**
- 用户回答后仍有模糊之处，继续追问直到清晰
- 四个维度全部覆盖后，**回述需求给用户确认**，不得假设用户已同意

## 全局规则

### 失败处理策略

| 规则 | 说明 |
|------|------|
| 阶段内重试上限 | 同一问题的修复尝试不超过 **3 次** |
| 阶段回退上限 | 同一阶段最多执行 **2 轮** |
| 超过上限 | 立即中止，生成中止报告（格式见下） |
| 编译失败 | 不得跳过，必须在当前任务内修复 |
| 测试失败 | TDD 红灯阶段正常现象，绿灯前不得进入下一任务 |

**中止报告标准格式** — 保存至 `$REPORT_DIR/abort-report.md`：

```markdown
# 工作流中止报告

## 中止原因
- 阶段：<N> / 7
- 原因：重试3次仍失败 / 阶段回退2轮仍失败 / 用户主动中断
- 失败详情：<具体错误信息>

## 已完成的工作
| 阶段 | 状态 | 产出文件 |
|------|------|---------|

## 已尝试的方案
1. 方案A: <描述> → <失败原因>
2. 方案B: <描述> → <失败原因>

## 建议的人工介入方向

## 恢复指引
如需恢复，在新会话中执行：从阶段<N>开始
参考文件：$REPORT_DIR/workflow-progress.md
```

**修复操作权限边界：**
- 编译/测试失败时，只允许修改当前任务涉及的、报错位置所在的文件
- **禁止**自动安装系统级依赖。如需安装，须向用户说明理由并获确认
- 修改构建配置文件（Makefile、CMakeLists.txt、.csproj 等）前，须向用户展示 diff 并获确认

### 证据强制规则

**所有检查点不接受 AI 口头声明，必须附带工具执行的实际输出作为通过证据：**

| 检查点类型 | 证据要求 |
|-----------|---------|
| Shell 命令 | 贴出 exit code 和关键输出行 |
| 文件读取/搜索 | 贴出读取到的文件内容或搜索结果 |
| 匹配搜索 | 贴出匹配行和匹配数量 |

- 不得使用"已通过"、"看起来没问题"、"测试全部绿灯"等未经证实的表述
- 如工具输出为空或异常，视为未通过

### Git 版本控制

每完成一个阶段（以及阶段3 中每完成一个子任务），执行一次 commit。

commit message 规范：
```
dev-workflow: 阶段<N> <阶段名> — <产出摘要>
```

安全提交流程：
```bash
git status --short
# 排除敏感文件：.env / *.key / secrets / credentials* / *.pem / *.pfx
git add --all -- ':!.env' ':!*.key' ':!secrets' ':!credentials*' ':!*.pem' ':!*.pfx'
git diff --cached --stat
git commit -m "<message>"
```

### Context 管理策略

1. 每完成一个阶段，将阶段产出写入文件
2. 进入下一阶段时，读取上阶段文件的关键部分（前 200 行）
3. 阶段3 每个子任务完成后，输出摘要（≤50 行）写入 `$REPORT_DIR/task-summary.md`
4. 如果预计 context 将超出限制，将当前状态写入 `$REPORT_DIR/workflow-progress.md`，**建议用户开启新会话并从对应阶段恢复**

### 可观测性（Telemetry）

每个阶段的开始/结束时，更新 `$REPORT_DIR/workflow-telemetry.json`：

## 流程总览

```
阶段1: 规划与设计   → 苏格拉底追问 → 需求分析 → 方案对比 → 实现计划
阶段2: 架构审查     → 8维度评分 → 问题清单 → 替代路线推荐
阶段3: TDD 编码     → 语言检测 → 微任务拆分 → 红绿重构循环 → 编译验证
阶段4: 系统化调试   → 复现→定位→根因→修复→验证（处理残留复杂bug）
阶段5: 代码审查     → 5维度逐项检查 → 修改任务表 → 逐项修复
阶段6: 代码优化     → 简洁性+可读性+可维护性 → Lint 检查
阶段7: 文档生成     → README + 工作流程报告 + 使用手册 + 移植方案 + 遥测
```

## 阶段1：规划与设计

### Step 1.1 · 苏格拉底追问（如 Scope Gate 中未完成则补做）

依次追问 4 个维度，回述需求给用户确认后方可进入下一步。

### Step 1.2 · 需求分析与方案对比

根据确认的需求，探索 **2-3 种**技术方案，每种方案说明：
- 核心思路（1-2 句）
- 优点（2-3 条）
- 缺点（2-3 条）
- 推荐方案及理由

输出设计文档到 `$DESIGN_DIR/YYYY-MM-DD-<topic>-design.md`：

```markdown
# <项目名> 设计文档

## 需求概述
<经苏格拉底追问确认后的需求总结，2-3段>

## 方案对比
| 方案 | 核心思路 | 优点 | 缺点 | 推荐 |
|------|---------|------|------|------|

## 技术栈与依赖
| 组件 | 版本 | 用途 |
|------|------|------|

## 模块划分
| 模块 | 职责 | 对外接口 |
|------|------|---------|

## 接口设计
<关键函数签名、数据模型、通信协议>

## 数据流
<输入→处理→输出的文字描述，标注关键转换节点>

## 验收标准
| # | 标准 | 验证方式 |
|---|------|---------|

## 风险与约束
| 风险 | 影响 | 缓解措施 |
|------|------|---------|
```

### Step 1.3 · 实现计划

将设计分解为有序任务列表，输出到 `$DESIGN_DIR/YYYY-MM-DD-<topic>-plan.md`：

```markdown
# 实现计划

## 任务列表
| # | 任务描述 | 目标文件 | 预计行数 | 依赖任务 |
|---|---------|---------|---------|---------|

## 执行顺序
<拓扑排序说明，标注可并行执行的任务组>

## 关键决策点
<实现过程中的关键决策和里程碑>
```

### 检查点

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 设计文档存在且 ≥200 字 | 读取 `$DESIGN_DIR/*-design.md` 前 30 行确认 |
| 2 | 👤 | 实现计划已获用户确认 | 向用户展示计划摘要，等待确认回复 |
| 3 | 🔧 | 技术栈和依赖已明确 | 检查设计文档含「技术栈」或「依赖」章节 |

全部通过 → git commit → 更新 telemetry → 进入阶段2

## 阶段2：架构审查

### 审查流程

独立分析设计文档（不修改任何文件），从 8 个维度评分：

### 评分维度与标准

| 维度 | 5星标准 | 1星标准 |
|------|--------|--------|
| **模块划分** | 职责单一，边界清晰 | 职责混乱，边界模糊 |
| **接口设计** | 耦合度可控，语义清晰 | 紧耦合，接口隐式依赖 |
| **数据流** | 流向清晰，无循环依赖 | 存在循环依赖或数据回流 |
| **可扩展性** | 新增功能不改核心代码 | 任何改动都需大规模修改 |
| **可测试性** | 每模块可独立测试 | 无法进行单元测试 |
| **可移植性** | 平台相关代码完全隔离 | 强平台依赖，难以移植 |
| **性能** | 无明显瓶颈 | 存在严重性能问题（O(n²)以上） |
| **安全性** | 无常见漏洞（注入/越界/泄漏） | 存在严重安全漏洞 |

每个评分必须附 1-2 句评分理由。

### 输出格式

审查报告写入 `$REPORT_DIR/architecture-review.md`：

```markdown
# 架构审查报告

## 总体评价
<2-3句总结>

## 各维度评分
| 维度 | 评分(1-5) | 评分理由 |
|------|----------|---------|

## 发现的问题
| 编号 | 问题描述 | 严重程度 | 影响范围 |
|------|---------|---------|---------|
| P1 | <描述> | 严重/一般/建议 | <影响> |

## 改进建议
<按优先级排序，每个建议附 1-2 句说明>

## 替代技术路线（可选）
<1-2条，含适用场景、优劣对比、迁移成本>

## 结论
<1-2句总结>
```

### 检查点

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 审查报告含全部章节（7个顶级标题） | 读取前 80 行验证 |
| 2 | 🔧 | 问题编号列表完整（含编号列） | 检查问题表格结构 |

全部通过 → git commit → 更新 telemetry → 进入阶段3

## 阶段3：TDD 编码

### 3.1 项目语言检测

扫描项目目录，自动选择构建和测试命令：

| 特征文件 | 语言 | 构建命令 | 测试命令 |
|---------|------|---------|---------|
| `*.csproj` / `*.sln` | C# | `dotnet build` | `dotnet test` |
| `Makefile` + `*.c` | C (Make) | `make -j4` | `make test` |
| `CMakeLists.txt` + `*.c` | C (CMake) | `cmake --build build/` | `ctest --test-dir build/` |
| `CMakeLists.txt` + `*.cpp` | C++ | `cmake --build build/` | `ctest --test-dir build/` |
| `package.json` + `*.ts` | TypeScript | `npx tsc --noEmit` | `npm test` |
| `package.json`（仅 JS） | JavaScript | `npm run build` | `npm test` |
| `go.mod` | Go | `go build ./...` | `go test ./...` |
| `Cargo.toml` | Rust | `cargo build` | `cargo test` |
| `pom.xml` | Java Maven | `mvn compile` | `mvn test` |
| `build.gradle*` | Java Gradle | `gradle build` | `gradle test` |
| `pyproject.toml` | Python | `python -m py_compile <files>` | `pytest` |
| `STM32*.ioc` | STM32 | `STM32CubeIDE ... -build <project>` | 编译检查+静态分析 |

用户手动指定的构建命令优先。

### 3.2 微任务拆分（2~5 分钟粒度）

将实现计划拆分为微任务，每个任务必须能在 **2~5 分钟**内完成。

**每个微任务必须包含：**
- **精确文件路径** — 如 `src/auth/login.ts（新建）`，不得写"相关文件"
- **完整实现步骤** — 如"1. 定义 LoginRequest 接口… 2. 实现 handleLogin()…"
- **验证命令** — 如 `npx jest src/auth/login.test.ts`，不得写"运行测试验证"

**拆分优先级：** 数据模型/接口 → 核心逻辑 → UI/集成 → 互不依赖的可并行

**示例：**
```
微任务 3/7: 创建 POST /login 路由（预计 3 分钟）
  → 文件: src/auth/login.ts（新建）, src/auth/router.ts（修改）
  → 步骤:
    1. src/auth/login.ts: 定义 LoginRequest 接口（username, password）
    2. src/auth/login.ts: 实现 handleLogin(req): Promise<LoginResponse>
    3. src/auth/router.ts: 注册 POST /login 路由
  → 验证: npx jest src/auth/login.test.ts
```

### 3.3 TDD 循环（每个微任务内执行）

```
① 写失败测试
      ↓
② 运行测试（必须是红灯 — 测试失败）
      ↓ 红灯 ✓
③ 写最少实现代码（刚好让测试通过）
      ↓
④ 运行测试（必须绿灯）
   ├─ 绿灯 → ⑤
   └─ 失败 → 回到③（最多3次）
      ↓
⑤ 重构（不改外部行为，保持测试绿灯）
      ↓
⑥ 运行全部测试（确保无回归）
   ├─ 全部绿灯 → 任务完成
   └─ 有失败 → 回到③
```

**约束：**
- 未写测试前不得写实现代码
- 红灯才能写实现，绿灯立即停止写实现
- 重构不得改变外部行为
- 每个任务完成后运行全量测试

### 3.4 无测试环境的降级方案

**如果项目无法写自动化测试（如嵌入式裸机 C）：**

用编译检查 + 静态分析替代，必须执行以下工具中的至少一种：

| 语言 | 静态分析命令 |
|------|------------|
| C/C++ | `cppcheck --enable=all --inconclusive --std=c11 .` |
| C# | `dotnet format --verify-no-changes` |
| Python | `ruff check .` 或 `mypy .` |

- 在任务摘要中标注"无自动化测试，已通过编译+<工具名>"
- TDD 循环简化为：写实现 → 编译 → 静态分析 → 通过 → 下一任务

### 3.5 编译验证（每个任务后强制执行）

```bash
<根据 3.1 检测到的构建命令>
```

通过条件：exit code == 0 且无 `error:` 级别输出。

**嵌入式烧录验证（可选）：**
- ST-Link: `st-flash write <firmware.bin> 0x08000000`
- OpenOCD: `openocd -f board/stm32f103c8t6.cfg -c "program <firmware.elf> verify reset"`

### 3.6 编译/测试失败处理

```
失败 → 读取错误输出 → 定位问题 → 修复(仅修改报错文件) → 重新编译/测试
                                              ↓
                                        仍失败(第2次)
                                              ↓
                                        换方案 → 重新编译/测试
                                              ↓
                                        仍失败(第3次)
                                              ↓
                              简单bug？→ 进入阶段4 系统化调试
                                              ↓
                              无法修复？→ 生成中止报告
```

### 检查点（每个微任务后）

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 测试绿灯 或 编译+静态分析通过 | Shell 执行测试/构建命令，exit code 0 |
| 2 | 🔧 | 无新增未处理 warning | 检查构建输出，写入 `$REPORT_DIR/build-log.md` |
| 3 | 🔧 | 接口一致性 | 搜索新接口引用，抽样验证前 5 个 |
| 4 | 🔧 | 构建日志已保存 | 确认 `$REPORT_DIR/build-log.md` 含本次记录 |

每个微任务通过 → git commit → 下一个微任务

### 3.7 阶段3→4 桥接检查

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 所有微任务 TDD 循环已闭合 | 读取 `$REPORT_DIR/task-summary.md`，逐一确认标注通过 |
| 2 | 🔧 | 构建日志无未处理失败 | 检查无残留 `error:` 或 `FAILED` |
| 3 | 🔧 | 全量测试/构建最后一次通过 | Shell 执行全量测试，exit code 0 |
| 4 | 🔧 | 无 TODO/FIXME 残留（或已记录） | 搜索 `TODO\|FIXME\|HACK\|XXX` |

全部通过 → 写入摘要到 `$REPORT_DIR/task-summary.md` → 更新 telemetry → 进入阶段4

## 阶段4：系统化调试

### 适用条件
- 阶段3 中 3 次重试内无法修复的 bug
- 阶段5（代码审查）中发现的复杂逻辑错误
- 用户主动触发的调试请求

**无待调试问题 → 跳过本阶段，直接进入阶段5。**

### 调试流程（7步）

```
① 复现 bug
   记录复现步骤和预期行为 vs 实际行为
      ↓
② 缩小范围（二分法定位）
   注释一半代码 → 运行测试 → 哪一半报错？→ 继续二分直到定位
      ↓
③ 分析根因
   连续追问：为什么这行代码会导致这个问题？→ 为什么这个条件会触发？→ …直到找到根因
      ↓
④ 设计修复方案
   最小改动原则 — 只修改必要的代码，不顺手重构
      ↓
⑤ 实施修复
      ↓
⑥ 验证修复 + 回归测试
   原复现步骤不再触发 + 相关测试全部通过 + 无新增 warning
      ↓
⑦ 记录到 $REPORT_DIR/bug-fix-log.md
```

### 检查点

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 所有待调试 bug 已处理 | 读取阶段3 失败记录，逐一确认有修复日志 |
| 2 | 🔧 | 修复后全量测试通过 | Shell 执行全量测试，exit code 0 |

全部通过（或跳过）→ git commit → 更新 telemetry → 进入阶段5

## 阶段5：代码审查

### Step 5.1 · 执行审查

对照以下 5 个维度逐项检查代码，对照阶段2 的问题列表逐一验证已解决：

| 审查维度 | 检查内容 |
|---------|---------|
| **功能完整性** | 是否实现了设计文档中的所有需求？所有验收标准是否满足？ |
| **逻辑正确性** | 边界条件是否处理？空指针/null 检查？整数溢出？死循环？竞态条件？ |
| **安全性** | SQL/命令注入？缓冲区越界？敏感信息泄漏？权限检查缺失？ |
| **风格规范** | 命名是否一致？注释是否必要的？文件组织是否符合项目约定？ |
| **性能** | 不必要的拷贝/分配？N+1 查询？未释放的资源/文件句柄？ |

输出修改任务表到 `$REPORT_DIR/code-review-issues.md`：

```markdown
# 代码审查修改任务表

| # | 文件 | 行号 | 问题描述 | 严重程度 | 修改方案 | 状态 |
|---|------|------|---------|---------|---------|------|
| 1 | src/foo.c | 42 | 未检查malloc返回值 | 严重 | 添加NULL检查和错误返回 | 待修复 |
```

严重程度：`严重`（必须修）/ `一般`（应该修）/ `建议`（可选修）

### Step 5.2 · 逐项修复

1. 严重 → 一般 → 建议，按优先级逐个修复
2. 每修复一项 → 编译验证 → 更新状态为「已修复」
3. 3 次重试上限，仍失败则记录到中止报告
4. 复杂 bug 可回到阶段4 流程处理

### 检查点

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 无未关闭的严重/一般问题 | 逐行读取任务表确认 |
| 2 | 🔧 | 建议类问题已逐条评估（标注采纳/不采纳原因） | 逐条检查 |
| 3 | 🔧 | 修复后编译+测试通过 | Shell 执行构建+测试，exit code 0 |
| 4 | 🔧 | 阶段2 问题全部回溯验证 | 对照架构审查报告问题列表逐个标记 |

全部通过 → git commit → 更新 telemetry → 进入阶段6

## 阶段6：代码优化

### 优化维度

| 维度 | 具体操作 |
|------|---------|
| **简洁性** | 删除死代码；合并重复逻辑（≥3次重复→提取）；简化过度复杂的条件表达式；移除无用抽象层 |
| **可读性** | 变量/函数命名为业务语义；复杂逻辑加一行 why 注释；统一代码风格；合理使用空行分组 |
| **可维护性** | 提取魔法数字为命名常量；降低圈复杂度（if嵌套>3→提取函数）；减少隐式全局状态 |

| # | 类型 | 验证内容 | 操作 |

| 2 | 🔧 | 全量测试通过 | Shell 测试命令，exit code 0 |
| 3 | 🔧 | Lint 通过（无 error 级别问题） | 执行对应 lint 命令 |

全部通过 → git commit → 更新 telemetry → 进入阶段7
